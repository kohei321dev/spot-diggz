# Discovery Sprint 0 検証計画

- Status: Approved for validation
- Date: 2026-07-14
- Target area: 大阪市・堺市を利用者の起点とする大阪都市圏
- Primary persona: 都市部でスケートボードを始めた初心者
- Related issue: GitHub Issue #278

## 1. Sprint Goal

大阪市・堺市の初心者が、既存サービスだけでは判断しにくい利用条件を使って、実際に滑りに行く施設を短時間で決められるか検証する。

このSprintはコード量ではなく、対象利用者、対象地域、成功条件、停止条件、施設候補、検証手順が揃い、次のDelivery Issueを開始できる状態を成果とする。

## 2. 検証する仮説

### Problem hypothesis

初心者は施設の住所だけでなく、現在営業中か、初心者が基礎練習できるか、60分以内で到着できるか、料金・登録・防具等のルールを同時に確認する必要があり、既存手段では行き先決定に手間がかかる。

### Solution hypothesis

目的、時間、出発地点、交通手段、初心者適性を入力し、検証済み施設を最大3件まで理由付きで提示すると、既存サービスより行き先を決めやすくなり、外部ナビと現地訪問につながる。

### Operation hypothesis

候補20〜30施設の動的情報を30日以内、安定情報を180日以内に確認しながら、週2時間以内で運用できる。

## 3. 対象者

### 必須条件

- 大阪市または堺市を日常の出発地点にできる
- スケートボードを始めて1年以内、または自分を初心者と認識している
- 4週間以内に少なくとも1回滑りに行く予定がある
- 自分で行き先を決めた経験、または決められず断念した経験がある

### 除外条件

- 地域の主要施設と利用条件を既に把握している中級・上級者
- 施設を探す予定がなく、仮想的な感想だけを答える参加者
- 未確認ストリートスポットの探索だけを目的とする参加者

### 人数

- Week 1の比較観察: 5〜8人
- Week 3からWeek 4の実利用: 合計10人以上

## 4. 4週間の検証

### Week 1: 既存手段の観察

参加者に、次回実際に行く施設をGoogle Maps、スケパ、OllieTown等の普段の手段で決めてもらう。

記録する項目:

- 探索開始から候補決定までの時間
- 使用したサービスと開いたページ数
- 比較した施設数
- 最後まで不明だった情報
- 行き先を決めた理由
- 外部ナビを開いたか
- 実際に訪問したか
- 行き先を決められず中断した理由

### Week 2: 手動データとプロトタイプ

- docs/discovery/osaka-facility-candidates.md の23候補をIssue #280で検証する
- 最初のプロトタイプは「今から行ける」「初心者向け」「60分以内」を優先する
- ログイン、投稿、独自地図、AI推薦は実装しない

### Week 3: 実利用

- 対象者へURLを渡し、実際の練習日に利用してもらう
- 条件入力、推薦結果、根拠表示、外部ナビ、訪問結果を記録する
- 既存手段を併用した場合は、何を確認するために戻ったか記録する

### Week 4: 再利用と継続判断

- 2回目以降の利用を確認する
- 施設情報の更新時間を合計する
- 成功条件と停止条件を判定する
- Go、仮説修正、No-GoのいずれかをSprint Reviewに記録する

## 5. インタビュー質問

「欲しい機能」ではなく、実際の行動と直近の失敗を確認する。

1. 最後に滑りに行った日は、どこへ行くかをどのように決めましたか。
2. 候補を決めるまでに、どのサービスや人を使いましたか。
3. 到着後に、営業時間、利用ルール、混雑、路面等が想定と違った経験はありますか。
4. 行きたいと思ったが、場所を決められず滑らなかった直近の例はありますか。
5. 今日の候補のうち、既存手段だけでは判断できなかった情報は何ですか。
6. 推薦された3件のうち、どれを選び、何が決め手になりましたか。
7. 推薦がなければ、最終的に何を使って決めていましたか。
8. 外部ナビを開いた後、実際に到着して滑れましたか。滑れなかった場合はなぜですか。
9. 4週間以内にもう一度使いましたか。使わなかった場合はなぜですか。
10. 表示された施設情報に誤りや不足はありましたか。

## 6. 計測定義

実装時のイベント名は安定した英小文字のsnake_caseとし、自由入力、正確な現在地、住所、利用者IDをメトリクスラベルへ保存しない。

| 指標 | 定義 | 目的 |
| --- | --- | --- |
| decision_started | 条件入力を開始したセッション数 | 利用開始を把握する |
| recommendation_completed | 1件以上の推薦結果を表示したセッション数 | 推薦完了率を測る |
| recommendation_empty | 推薦が0件だったセッション数 | 条件・データ不足を把握する |
| facility_detail_viewed | 施設詳細を表示したセッション数 | 比較行動を把握する |
| source_opened | 情報源を開いたセッション数 | 根拠確認を把握する |
| navigation_opened | 外部ナビを開いたセッション数 | 訪問意図を把握する |
| visit_confirmed | 利用者が到着・滑走できたと回答した件数 | 実行動を把握する |
| correction_submitted | 訂正報告を送信した件数 | データ品質を把握する |
| repeat_use | 同一参加者が4週間以内に2回以上利用したか | 継続性を測る |
| maintenance_minutes | 施設情報の確認・更新に使った分数 | 運用可能性を測る |

正確な現在地は推薦処理中だけ使用し、既定で保存しない。実利用検証で参加者を追跡する場合は、分析用IDと連絡先を分離し、目的、保持期間、削除方法を事前に説明する。

## 7. 継続条件

次をすべて満たす場合にGoとする。

- 10人以上が実際の行き先決定に利用する
- 5人以上が4週間以内に2回以上利用する
- 3人以上が既存サービスだけでは判断できなかった具体的な行動差を示す
- 固有条件を1つ以上、利用者の50%以上が使用する
- 5人以上が推薦結果から外部ナビを開く
- 3人以上が実際に到着して滑走できたと回答する
- 施設情報の確認・更新が週2時間以内に収まる

## 8. 停止条件

次のいずれかが確認された場合は追加実装を止め、仮説を再検討する。

- 外部ナビを開いた利用者が5人未満、または到着して滑走できた利用者が3人未満である
- 利用者が知人、Instagram、Google Mapsだけで最終判断し、推薦が行動を変えない
- 既存サービスより判断時間や不明点が改善しない
- 施設情報の確認・更新が週2時間を超える
- 検証済み施設が少なく、条件に合う候補を安定して返せない

## 9. Sprint Review

Sprint Reviewでは次を記録する。

- Sprint Goalを達成したか
- 継続条件ごとの実測値
- 停止条件に該当したか
- 利用者の行動が変わった具体例
- 情報が古い、候補がない等の失敗例
- Go、仮説修正、No-Goの判断と根拠

## 10. Retrospective

- 継続すること
- 問題だったこと
- 次に試すこと
