# Engineering Principles and Sources

- Status: Reference
- Date: 2026-07-12
- Normative rules: [AGENTS.md](../../AGENTS.md)

## 目的

ローカル所蔵の技術書から、このリポジトリへ採用した原則と、AGENTS.mdへ強制規則として反映した理由を記録する。
本文の転載ではなく、複数の書籍に共通する考え方を要約している。

## 採用基準

次を満たす原則をAGENTS.mdへ採用した。

- 特定のクラウドやフレームワークに依存しない
- レビュー時に違反を判定できる、またはCIで自動検証できる
- 個人開発でも維持可能である
- 品質、変更容易性、セキュリティ、信頼性のいずれかを直接改善する
- 現在のプロダクト要求と矛盾しない

## 書籍別の採用内容

### リーダブルコード

参照テーマ: 命名、コメント、制御フロー、変数、関数分割、テストの読みやすさ。

採用内容:

- コードを理解する時間を短くすることを可読性の基準にする
- 名前に対象、振る舞い、単位、状態を含める
- コメントはコードの言い換えではなく、理由と制約を説明する
- 深いネストと巨大な式を避け、制御フローを単純にする
- テストもプロダクションコードと同じ可読性基準で扱う

### 実践で学ぶコード改善の極意

参照テーマ: 小さなリファクタリング、カプセル化、依存の分離、コメントの整理、追加による変更。

採用内容:

- 変更を小さな検証可能単位で進める
- 変更前に仕様と安全網を確認する
- 状態と不変条件を局所化する
- 継承より合成を優先し、外部依存を境界へ隔離する
- コメントアウトしたコードは履歴へ任せ、現行ツリーに残さない

### ソフトウェアアーキテクチャの基礎 第2版

参照テーマ: モジュール性、凝集度と結合度、品質特性、適応度関数、ADR、トレードオフ分析。

採用内容:

- 技術選定を品質特性とトレードオフから説明する
- 高凝集・疎結合なモジュール境界を作る
- アーキテクチャ特性に測定・検証方法を持たせる
- 重要な判断をADRへ残し、却下案と悪影響も記録する
- 複雑な分散構成は、必要性を示す計測結果が出るまで採用しない

### 単体テストの考え方／使い方

参照テーマ: テストの価値、リファクタリングへの耐性、観察可能な振る舞い、モック、統合テスト。

採用内容:

- テストは実装詳細ではなく観察可能な振る舞いを検証する
- カバレッジ率を目的にせず、重要なドメインルールと失敗経路を優先する
- モックを外部境界へ限定し、内部呼び出し順序への結合を避ける
- 不具合を回帰テストへ変換する
- テストの保守コストを品質評価に含める

### 実践 AIエージェント開発

参照テーマ: ツール、メモリ、評価セット、監視、トレース、セキュリティ、人間との協働。

採用内容:

- AIシステムを確率的システムとして扱い、通常の単体テストだけで完了としない
- 評価セットを生きた仕様としてCIへ統合する
- ツール選択、引数、結果、レイテンシー、コストを測定する
- 本番の失敗トレースを評価ケースへ変換する
- 外部操作には最小権限と人間による確認を適用する
- プロンプトインジェクション、データ漏えい、ツール悪用を脅威として扱う

### 生成AIデザインパターン

参照テーマ: 構造化出力、RAG、グラウンディング、評価、キャッシュ、ガードレール。

採用内容:

- AI出力を構造化し、スキーマで検証する
- 事実回答を検証済み外部データへグラウンディングする
- 評価指標とプロダクト目的を区別する
- ルール、検索、生成、検証を分離する
- ガードレールを入力、実行、出力の複数層に置く

### 実践 ハイブリッドクラウド・セキュリティアーキテクチャ

参照テーマ: セキュア・バイ・デザイン、脅威モデリング、ゼロトラスト、最小権限、データ分類、監査。

採用内容:

- 新しい信頼境界とデータフローを追加する時点で脅威モデルを更新する
- ネットワーク位置を信頼根拠にせず、主体ごとに認証・認可する
- 通信中、保存中のデータを保護する
- データ分類に応じて保存、ログ、保持、削除を決める
- 脅威、管理策、検知方法を追跡可能にする

### 入門 監視

参照テーマ: ユーザー視点の監視、アプリケーション計測、構造化ログ、アラート、デプロイマーカー。

採用内容:

- 監視を担当者の役割ではなく、サービスを作る側のスキルと責任にする
- CPU等の原因候補より、ユーザーに現れる症状を先に監視する
- アプリケーション固有のビジネスメトリクスをコードへ組み込む
- 構造化ログを使い、検索可能なイベント名と相関IDを持たせる
- デプロイ情報をメトリクスへ重ね、変化との関係を追跡する
- アラートは行動可能なものに限定する

### SLO サービスレベル目標

参照テーマ: SLI、SLO、SLA、エラーバジェット、バーンレート、ユーザージャーニー。

採用内容:

- SLIはユーザー視点の良いイベントと悪いイベントで定義する
- SLOを契約ではなく、信頼性に関する意思決定の目標として扱う
- SLAは補償を伴う外部約定としてSLOと分離する
- エラーバジェットを機能開発と信頼性改善の判断材料にする
- SLOを一度決めて固定せず、計測結果と利用者要求に合わせて更新する

### 入門 継続的デリバリー

参照テーマ: バージョン管理、継続的インテグレーション、自動化、リリース可能状態、ロールバック。

採用内容:

- Gitをソフトウェアと設定のSingle Source of Truthにする
- 変更ごとに自動検証し、mainをリリース可能に保つ
- 小さな変更と早いフィードバックを優先する
- ビルド、設定、デプロイを再現可能にする
- 同一成果物を環境間で昇格させる
- ロールバックをリリース設計に含める

### システム運用アンチパターン / SREの探求

参照テーマ: 盲目運用、アラート疲れ、自動化、インシデント指標、非難のないポストモーテム。

採用内容:

- アラート数ではなく、検出・対応・復旧の有効性を測る
- 反復的な手作業を自動化し、運用負荷を追跡する
- インシデントを再発防止の入力として扱う
- ポストモーテムは個人の責任追及ではなく、システムとプロセスの改善に使う
- 信頼性100%を目標にせず、プロダクト価値とのトレードオフを管理する

## 採用しなかった内容

- 特定製品やクラウドサービスを前提とする実装例
- 現在の規模に対して過剰なマイクロサービス、Kubernetes、複数クラウド構成
- 根拠のない一律のカバレッジ率、SLO値、ログ保持期間
- 監視データを無制限に収集する方針
- AIに全判断を委ねる完全自律構成

これらは要件、規模、実測値が揃った時点でADRにより再評価する。
