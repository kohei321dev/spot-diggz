# ADR-0002 施設データの出典と鮮度管理

- Status: Accepted
- Date: 2026-07-14
- Geographic scope partially superseded by: [ADR-0011](0011-five-prefecture-mvp-scope.md)

## Context

推薦サービスは、施設の住所だけでなく、営業時間、料金、利用可能競技、初心者適性、照明、屋根、防具ルールなどを必要とする。

スケート施設の利用可否や営業時間は変化するため、未確認の情報を推薦すると、利用者が現地で滑れない、ルール違反になる、サービスを信頼しなくなる可能性がある。

## Decision

以下のデータ方針を採用する。

1. 初期利用者の起点と施設候補の地理scopeはADR-0011を正とする。
2. 施設候補は5府県の合計20〜30件を目標とする。移動可否は施設の固定属性にせず、出発地点、時刻、交通手段、利用可能時間ごとに判定する。
3. 候補台帳への登録と公開対象への採用を分ける。公開前に利用可否、住所、営業時間、料金、初心者適性、ルール、情報源を確認する。
4. 自治体・施設の公式ページ、公式案内、利用規約を優先する。業界団体やディレクトリは候補発見に利用できるが、単独では公開根拠にしない。
5. 施設ごとに `sourceUrl`、`sourceType`、`verifiedAt`、`dynamicVerifiedAt`、`stableVerifiedAt`、`status` を保存する。確認できない属性は推測せず、公開必須属性にしないか候補のまま保持する。
6. 営業時間、料金、予約、休場、利用ルール等の動的情報は30日以内、名称、住所、地域、設備、アクセス等の安定情報は180日以内を初期鮮度期限とする。
7. `dynamicVerifiedAt` または `stableVerifiedAt` が期限を超えた施設はcatalog参照には残すが、推薦から除外する。未来の検証時刻は起動時に拒否する。
8. 休場期間は、一回限りの `one_time` を `YYYY-MM-DD`、毎年繰り返す `annual` を `MM-DD` の開始・終了日で構造化する。両端を含み、年をまたぐ `annual` も扱う。
9. 利用可否不明のストリートスポットはMVPに含めない。
10. Instagramや検索結果からの無断スクレイピングを前提にしない。公式SNSしか存在しない場合は候補発見に留め、公開方法を個別確認する。
11. 20〜30施設の確認・更新作業は週2時間を運用上限とする。4週間の検証で上限を超えた場合は、施設数、属性、鮮度期限の順に縮小を検討する。
12. 全国展開は、初期検証とデータ更新手順が成立した後に判断する。
13. Milestone 1ではAIを使用しない。将来導入する場合も、AIはデータを新規生成せず、検証済みデータの検索・説明・翻訳に限定する。

## Alternatives

### 557施設を最初に一括登録する

全国価値を早く表現できるが、情報確認と更新の負担が大きく、検証前にデータ作成で停止する可能性が高い。

### ユーザー投稿だけでデータを作る

投稿コストを分散できるが、コールドスタート、誤情報、違法スポット、秘密スポットの公開、モデレーションが問題になる。

### Google Maps等の外部情報だけを利用する

住所・経路には使えるが、スケートボードのレベル、路面、セクション、ルール、初心者適性が不足する。

## Consequences

### Positive

- 推薦理由の根拠を説明できる
- 施設情報の更新責任を明確にできる
- AIの誤回答を抑制できる
- 利用者が情報の鮮度を判断できる

### Negative

- 初期データの手動作成が必要になる
- 全国展開まで時間がかかる
- 施設ごとに情報の粒度が揃わない可能性がある
- 時刻、起点、交通手段、利用可能時間によって推薦対象が変わる
- 30日以内の動的情報確認が週2時間の上限に収まらない可能性がある
- 5府県へscopeを広げることで確認・英訳の運用負荷が増える

## Open Questions

- 訂正報告をcatalogへ反映するreview ownerとSLAをどう定義するか
- 民間施設の掲載条件と、写真・ロゴ・説明文の利用可否をどのように確認するか
- 施設情報の英訳を誰が確認するか
- 30日 / 180日の確認作業を週2時間以内で維持できるか

## Verification

- 5府県の公開catalogが20〜30件の目標と重複しないIDを持つこと
- 公開対象は一次情報による必須属性確認を完了していること
- 動的情報と安定情報を `dynamicVerifiedAt` / `stableVerifiedAt` で別々に判定できること
- 30日 / 180日の期限超過施設が推薦から除外され、metricsでstaleとして観測できること
- `one_time` / `annual` の休場期間と年またぎをtestできること
- 週次の確認・更新時間を計測し、2時間以内かSprint Reviewで確認すること

## Evolution 2026-07-20: 鮮度と休場の実装形式

### Final Goal (unchanged)

検証済みの新しい施設情報だけを推薦判断へ使い、利用者が現地で滑れないriskを下げる。

### Means / Implementation (changed)

| 論点 | ADR確定時 | 実装変更後 | ゴール影響 |
| --- | --- | --- | --- |
| 鮮度時刻 | 動的・安定情報を別判定する方針 | `dynamicVerifiedAt` / `stableVerifiedAt` を必須化 | 安全側に強化 |
| 期限超過 | 断定に使わない | catalog APIには残し、推薦から除外してmetrics化 | 調査可能性を維持 |
| 休場 | 自由文の休業情報 | `one_time` / `annual` の日付範囲を追加 | 決定論的判定を可能化 |
| 未来時刻 | 未定義 | 起動時loadで拒否 | 誤った鮮度判定を防止 |

### Why Changed

- [事実] 推薦engineは到着日の休場と、30日 / 180日の両方を機械判定する必要がある。
- [事実] 自由文の `scheduleNotes` だけでは、年末年始や一時休場を除外条件へ安全に変換できない。
- [判断] 表示用の注意文を残しつつ、除外に使う値を型付けしたfieldへ分離する。
