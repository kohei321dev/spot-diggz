# ADR-0001 リポジトリと旧実装の扱い

- Status: Accepted
- Date: 2026-07-12

## Context

spot-diggzには、旧スケートスポット検索構想のiOS、Android、Web、Rust API、GCP関連実装が存在する。

新しいプロダクトは、AIを使ったスケートセッション意思決定支援Webサービスとして再定義する。一方、旧実装には将来再利用できるコード、設計履歴、学習資産が含まれている。

## Decision

1. 既存リポジトリを継続利用する。
2. 既存の`master`は現在の安定状態として保持する。
3. 方向転換時点の退避として`archive/pre-session-planner-20260712`を作成する。
4. 新プロダクトは`product/session-planner`で開発する。
5. 旧iOS、Android、Rust API、MapKit、CloudKit等は、初期段階では削除しない。
6. GitHubリポジトリ名は、プロダクト検証が完了するまで変更しない。

## Alternatives

### 新規リポジトリを作成する

履歴を分離しやすいが、既存のIssue、CI、ドキュメント、設計履歴との関係が切れる。現時点では、旧実装と新サービスの関係がまだあるため採用しない。

### masterから旧実装を削除する

新サービスの見通しはよくなるが、復旧・比較・再利用が難しくなる。プロダクト価値が未検証の段階では採用しない。

### リポジトリ名を先に変更する

ブランド変更を先行できるが、URL、CI、Issue、外部リンクの更新が発生する。サービス名称が確定してから実施する。

## Consequences

### Positive

- 旧実装を安全に保持できる
- 新サービスの作業範囲をブランチで明確に分離できる
- 既存のGitHub運用とCIを再利用できる
- 方向転換前後の設計判断を比較できる

### Negative

- リポジトリ内に旧実装と新実装が一時的に共存する
- READMEやAGENTS.mdに旧方針が残るため、現行方針を明示する必要がある
- 将来、旧実装をアーカイブ専用リポジトリへ移す判断が必要になる可能性がある

## Follow-up

- `docs/product_baseline.md`を新しい要求の基準にする
- 旧実装を変更する場合は、目的と影響をIssueに記録する
- 新プロダクトが検証できた後に、旧実装の削除または別リポジトリ化を再評価する
